# Формулировка проекта

Этот документ фиксирует рабочую постановку проекта как развитие идей, изложенных в `TZ.docx` и `prompt.txt`. Он не заменяет исходные документы, а приводит их к одной операционной формулировке для реализации MVP и дальнейшего развития системы.

## 1. Главная идея

Проект строится вокруг pattern-first представления погоды.

Мы не рассматриваем прогноз как прямое предсказание сырых рядов `temp`, `rain`, `msl`, `wdsp` и других наблюдаемых величин. Вместо этого:

1. сырые погодные временные ряды используются как наблюдаемые каналы;
2. из них извлекаются сглаженные сигналы, производные, экстремумы, пики и hazard-функционалы;
3. на этой основе строится поток структурных событий;
4. из окон этого потока строятся представления паттернов;
5. модель учится предсказывать будущую последовательность паттернов на заданном временном интервале;
6. затем эта последовательность декодируется обратно в прогноз значений, временных характеристик, пиков и опасных событий.

Ключевой принцип:

- паттерн первичен;
- значения являются value placeholders;
- время является time placeholders;
- итоговый прогноз есть результат декодирования будущего pattern flow.

## 2. Объект моделирования

Базовый датасет: `hly4935_subset.csv`.

Рабочая гипотеза проекта состоит в том, что погодная динамика лучше представляется не как непрерывный поток чисел, а как поток повторяющихся структурных конфигураций, возникающих в нескольких каналах одновременно и отличающихся:

- направлением изменений;
- локальными экстремумами и точками перегиба;
- межканальными зависимостями;
- временной организацией событий;
- наличием или отсутствием peak/hazard-сценариев.

Следовательно, объектом предсказания является не одна числовая траектория сама по себе, а эволюция паттернов на будущем отрезке времени.

## 3. Каналы и placeholders

### 3.1. Value placeholders

Основные каналы MVP:

- `temp` -> `temperature`
- `rhum` -> `relative_humidity`
- `msl` -> `pressure`
- `wdsp` -> `wind_speed`
- `rain` -> `rainfall`

Дополнительные каналы:

- `dewpt`
- `wetb`
- `vappr`
- `wddir`

### 3.2. Time placeholders

Временная структура не считается внешней метаинформацией. Она входит в само представление паттерна.

В MVP time placeholders должны включать как минимум:

- шаг дискретизации `dt`;
- длину окна `L`;
- длину прогнозного интервала;
- интервалы между экстремумами;
- ширины пиков;
- длительности превышения порогов;
- средние и дисперсионные характеристики межсобытийных лагов;
- нормированные позиции событий внутри окна, где это полезно.

## 4. Операционный pipeline

Рабочая схема проекта:

`raw data -> cleaned data -> smoothed signals -> derivatives -> extrema stream -> extrema windows -> pattern windows -> pattern discovery -> pattern flow dataset -> future pattern sequence prediction -> decode values/time/peaks/hazards`

Это означает:

1. сначала строится событийно-структурное представление;
2. затем на нем выделяются повторяющиеся паттерны;
3. после этого формируется supervised-задача в пространстве паттернов;
4. только последним шагом выполняется декодирование в привычный прогноз погоды.

## 5. Границы и события

Границы паттернов определяются через точки перегиба и экстремальные события.

Для каждого канала после сглаживания вычисляются:

- первая дискретная производная;
- вторая дискретная производная;
- локальные экстремумы;
- пики как объекты с формализованными характеристиками;
- hazard-функционалы на окнах.

Единичное событие в общем потоке должно иметь как минимум:

- время;
- канал;
- тип события;
- знак;
- амплитуду;
- связанные значения производных;
- локальные временные характеристики.

После этого события всех каналов объединяются в единый `extrema stream`.

## 6. Pattern representation

Окно паттерна является составным объектом, а не одним вектором признаков.

Для каждого окна строится по крайней мере:

- `intra_matrix`: внутриканальная структура;
- `inter_matrix`: межканальные зависимости;
- `peak_hazard_matrix`: слой пиков и опасных событий;
- `time_placeholders`: временные характеристики окна;
- `feature_vector`: плоское представление для discovery и baseline-моделей.

Таким образом, `feature_vector` считается производной технической формой, а не полным смысловым описанием паттерна.

## 7. Pattern discovery

После построения окон паттернов система должна искать повторяющиеся конфигурации.

Цель discovery-слоя:

- присвоить окнам `pattern_id` или embedding;
- получить прототипы или центры паттернов;
- превратить последовательность окон в последовательность состояний pattern flow.

На уровне MVP допустим baseline через `k-means` или иерархическую кластеризацию. В дальнейшем этот слой должен оставаться сменным.

## 8. Строгая постановка supervised-задачи

Это ключевое уточнение относительно исходных формулировок.

### 8.1. Что именно предсказывает модель

На инференсе система должна предсказывать не один будущий паттерн, а будущую последовательность паттернов на заданном временном отрезке.

Следовательно, правильная формулировка задачи:

`history of patterns + forecast interval specification -> future pattern sequence`

а не:

`history of patterns -> single next pattern`

### 8.2. Что такое прогнозный интервал

Прогнозный интервал задает будущий временной отрезок, на котором требуется получить pattern flow. Он может быть выражен через:

- число будущих шагов;
- длину временного промежутка;
- число будущих pattern windows;
- комбинацию горизонта и stride.

Для MVP допустима фиксированная настройка, например 24 часа, но архитектура не должна считать это единственно возможным вариантом.

### 8.3. Каким должен быть target обучения

`ForecastSample` должен содержать:

- историю наблюдаемых или уже размеченных паттернов;
- спецификацию прогнозного интервала;
- целевую будущую последовательность `pattern_id`, `pattern vectors` или обе формы одновременно.

Иначе говоря, supervised dataset должен быть sequence-to-sequence или sequence-to-future-segment, а не single-label dataset.

### 8.4. Что возвращает инференс

`ForecastResult` должен содержать:

- время начала прогноза;
- спецификацию прогнозного интервала;
- предсказанную последовательность идентификаторов паттернов, если используется дискретное представление;
- предсказанную последовательность векторов или объектов паттернов;
- декодированный прогноз значений по каналам на всем интервале;
- декодированные time placeholders на всем интервале;
- декодированные пики и hazard events на всем интервале.

## 9. Декодирование

Декодирование должно работать не с одной точкой прогноза, а с будущим сегментом pattern flow.

Это означает наличие как минимум трех логических декодеров:

- decoder значений;
- decoder time placeholders;
- decoder peaks/hazards.

Каждый из них получает на вход будущую последовательность паттернов и строит прогноз на всем целевом интервале.

## 10. Что считается MVP

В рамках MVP система считается реализованной корректно, если она умеет:

1. загрузить и очистить `hly4935_subset.csv` с учетом quality flags;
2. построить сглаженные сигналы и производные;
3. детектировать экстремумы и пики;
4. сформировать окна экстремумов;
5. построить `PatternWindow` и его плоское представление;
6. выполнить baseline pattern discovery;
7. сформировать dataset для прогноза будущих последовательностей паттернов;
8. прогнать end-to-end pipeline до уровня структурных артефактов;
9. подготовить интерфейсы для последующего sequence predictor и декодеров.

MVP не обязан уже содержать сильную финальную sequence-модель, но обязан иметь корректную постановку данных и интерфейсы именно под предсказание будущего сегмента pattern flow.

## 11. Архитектурные следствия для кода

Из этой постановки следуют обязательные архитектурные требования:

- `PatternWindow` должен оставаться самостоятельным typed object, а не сводиться только к `numpy`-вектору;
- слой discovery должен быть отделен от слоя supervised forecasting;
- `ForecastSample` должен поддерживать target sequence;
- `ForecastResult` должен поддерживать sequence output;
- декодирование должно быть устроено как работа с последовательностью паттернов;
- `tau` и другие временные параметры должны быть частью представления данных, а не только внешними аргументами функции.

## 12. Требование к runtime моделей

Важное проектное ограничение:

- вся работа с моделями, то есть обучение и инференс, допускается только на GPU;
- CPU может использоваться для data pipeline, feature engineering, extrema/peak extraction, pattern discovery baseline и подготовки артефактов;
- model stage не должен молча деградировать в CPU fallback;
- если CUDA недоступна, training/inference entry point должен завершаться явной ошибкой конфигурации runtime.

Это требование нужно считать частью архитектуры, а не оптимизацией на будущее.

## 13. Итоговая рабочая формулировка

Итоговая постановка проекта в одной фразе:

Нужно построить MVP-систему, которая преобразует сырые почасовые метеорологические ряды в поток структурных погодных паттернов, выделяет повторяющиеся классы этих паттернов, формирует обучающие примеры по истории pattern flow и на инференсе предсказывает будущую последовательность паттернов на заданном временном отрезке с последующим декодированием в прогноз значений, временных характеристик, пиков и опасных погодных событий.
